# 性格雷达·生活指南答辩问答汇总

本文档用于软件工程大作业答辩前准备。回答口径尽量基于当前项目实际实现，避免把系统说成没有实现的 AI 或机器学习模型。

## 一、项目定位

### Q1：你们这个项目是做什么的？

A：项目名称是「性格雷达·生活指南」，它不是单纯的性格测试，而是把性格画像转化为生活决策建议的 Web 应用。用户完成性格、饮食、旅游和社交倾向测试后，系统会生成雷达图报告，并基于报告给出餐饮、旅游、社交推荐，还可以通过邀请码做双人适配分析。

完整流程是：

```text
首页介绍 -> 登录/注册 -> 完成测试 -> 生成报告 -> 查看推荐 -> 提交反馈 -> 双人适配 -> 个人中心管理数据
```

### Q2：为什么选择这个题目？

A：我们希望把一个常见的“性格测试”做成更完整的软件产品。普通测试往往只给一个结论，但用户不知道结果有什么用。我们的重点是把测试结果用于生活场景，比如吃什么、去哪玩、适合什么社交方式、两个人相处需要注意什么，这样更贴近真实产品。

### Q3：项目解决了什么问题？

A：主要解决三个问题：

- 用户不了解自己的生活偏好，只能凭感觉做选择。
- 普通测试缺少解释性，用户看完分数不知道下一步做什么。
- 双人适配如果直接展示完整答题会有隐私问题，所以我们只展示维度差异和相处建议。

## 二、功能模块

### Q4：系统有哪些核心模块？

A：主要有八个模块：

- 首页：展示产品定位、流程和入口。
- 登录注册：支持默认演示账号和管理员账号。
- 测试中心：完成性格、饮食、旅游、社交四类测试。
- 报告页：展示雷达图、画像标题、维度解释和推荐依据。
- 推荐页：展示餐饮、旅游、社交推荐，并支持喜欢、一般、不喜欢反馈。
- 双人适配：通过邀请码授权，生成双人画像对比和相处建议。
- 个人中心：管理测试记录、推荐反馈、匹配记录和隐私数据。
- 后台管理：管理用户、题库、推荐项和推荐规则。

### Q5：题库是怎么来的？

A：题库由两部分组成：

- 计划中原有的样例题库。
- `测试题.docx` 中的新题库内容。

当前前端题库共 `84` 题：

| 模块 | 题量 |
| --- | ---: |
| 性格测试 | 26 |
| 饮食偏好 | 20 |
| 旅游偏好 | 20 |
| 社交倾向 | 18 |

题目统一采用 1-5 级量表，便于计分和后续推荐计算。

### Q6：为什么要分成性格、饮食、旅游、社交四类测试？

A：性格维度是基础画像，用来理解用户的稳定倾向；饮食、旅游、社交是具体生活场景偏好。这样既有“人的长期性格”，也有“具体场景选择”，推荐会更贴近用户真实使用场景。

## 三、技术架构

### Q7：项目使用了哪些技术？

A：前端使用 Vue 3、Vue Router、Pinia、Vite、ECharts。后端使用 Spring Boot、Spring Security、Spring Data JPA。数据库和基础设施包括 MySQL、Redis、Docker Compose、Caddy 和 Cloudflare Tunnel。

### Q8：为什么前端选择 Vue 3？

A：Vue 3 适合快速构建交互型页面，组件化清晰，学习和维护成本较低。我们的页面包括测试流程、雷达图、推荐卡片、匹配报告、后台表格等，用 Vue 组件拆分比较自然。

### Q9：为什么后端选择 Spring Boot？

A：Spring Boot 适合课程项目中的标准后端开发，生态成熟，和 MySQL、Redis、Spring Security、JPA 结合方便。它也便于实现接口鉴权、用户管理、后台管理和后续部署。

### Q10：前后端是如何通信的？

A：前端通过 `/api` 调用后端接口。开发环境下 Vite 代理 `/api` 到 `http://127.0.0.1:8080`，正式临时发布时由 Caddy 把 `/api` 反向代理给后端服务。

### Q11：为什么还使用 localStorage？

A：产品化前端为了保证课程展示时即使后端接口不完整也能跑通闭环，把测试画像、推荐反馈、匹配记录等状态先封装在 service 层和 localStorage 中。这样 UI 和数据流可展示，后续替换成真实 API 时只需要改 service，不需要大改页面。

## 四、雷达图与计分算法

### Q12：你们生成雷达图用的什么算法？

A：雷达图不是 AI 生成，也不是机器学习模型，而是基于量表题的规则计分结果，再用 ECharts 的 radar 图表可视化。

计分流程如下：

```text
1. 每道题使用 1-5 分量表。
2. 如果是反向题，分数变成 6 - 原始分。
3. 把 1-5 分归一化到 0-100：
   normalized = round((value - 1) / 4 * 100)
4. 按维度分组求平均值。
5. 没有答题数据的维度默认 50 分。
6. 用五大性格维度绘制雷达图。
```

五大性格维度包括：

```text
开放性、尽责性、外向性、宜人性、情绪稳定性
```

### Q13：为什么要把 1-5 分转成 0-100？

A：0-100 更适合展示和比较。雷达图、推荐匹配度、双人契合度都使用百分制，用户更容易理解。例如 80 分比“4.2 分”更直观。

### Q14：反向题怎么处理？

A：反向题使用公式：

```text
反向分 = 6 - 原始分
```

例如一题描述的是“我经常临时决定重要安排”，它和尽责性是反方向的。如果用户选择 5 分，反向后就是 1 分，说明尽责性不高。

### Q15：雷达图里为什么展示“情绪稳定性”而不是“神经质”？

A：五大人格中常见维度有神经质，但“神经质”这个词对用户不够友好。我们在展示时转成“情绪稳定性”，含义更正向，也更容易被普通用户理解。

### Q16：画像标题是怎么生成的？

A：系统会比较五大性格维度的分数，选出最高维度，并映射成画像标题：

| 最高维度 | 画像标题 |
| --- | --- |
| 开放性 | 探索型生活家 |
| 尽责性 | 计划型生活家 |
| 外向性 | 社交型体验者 |
| 宜人性 | 温和型协调者 |
| 情绪稳定性 | 安静型观察者 |

## 五、生活推荐算法

### Q17：生活建议和推荐是怎么生成的？

A：推荐系统使用规则加权模型，不是黑盒 AI。每条推荐项会绑定相关的性格维度和场景偏好维度，系统根据用户画像计算匹配分，再结合历史反馈和推荐项基础分排序。

推荐得分公式是：

```text
推荐得分 = 性格匹配分 * 40%
        + 场景偏好分 * 40%
        + 历史反馈分 * 20%
```

代码中的权重是：

```text
personality = 0.4
preference = 0.4
feedback = 0.2
```

### Q18：性格匹配分怎么算？

A：每个推荐项会配置若干性格相关维度，例如开放性、外向性、宜人性。系统取用户这些维度分数的平均值，作为性格匹配分。

例如“创意融合小馆”可能更适合开放性和外向性高的用户，那么系统会读取用户的开放性和外向性分数，求平均作为性格匹配依据。

### Q19：场景偏好分怎么算？

A：场景偏好分来自饮食、旅游、社交等具体偏好维度。例如：

- 餐饮推荐会参考 `foodAdventure` 和 `foodSocial`。
- 旅游推荐会参考 `travelAdventure` 和 `travelPlanning`。
- 社交推荐会参考 `socialEnergy` 等维度。

系统同样对相关维度求平均。

### Q20：用户反馈如何影响推荐？

A：反馈会转成数值：

| 用户反馈 | 分数 |
| --- | ---: |
| 喜欢 | 100 |
| 一般 | 60 |
| 不喜欢 | 20 |
| 没有反馈 | 60 |

如果用户对某条推荐点“不喜欢”，该推荐项的反馈分下降，后续排序就会降低；如果点“喜欢”，相似内容会更容易排在前面。

### Q21：推荐项的最终匹配度为什么不是直接用公式分？

A：系统会把规则计算出的匹配分和推荐项自身的 `baseScore` 再平均一次。`baseScore` 表示推荐内容本身的基础质量或适用性，这样可以避免某些内容只因为维度匹配就被过度抬高。

最终会限制在 `1-99` 之间，避免出现绝对 0 或 100，表达上更稳妥。

### Q22：推荐理由是写死的吗？

A：不是完全写死。推荐说明由推荐项绑定的维度和用户画像动态生成。系统会找到该推荐项相关维度中用户得分最高的维度，再组合文案。

例如：

```text
因为你的开放性较高，愿意尝试新鲜体验，系统优先推荐「创意融合小馆」。
```

这样用户能知道“为什么推荐给我”，体现可解释性。

## 六、双人适配算法

### Q23：双人适配是怎么算的？

A：双人适配使用维度差异模型。系统不比较完整答题内容，只比较双方授权后的五大性格维度分数。

公式是：

```text
单维契合度 = 100 - abs(用户A维度分 - 用户B维度分)
总体契合度 = Σ(单维契合度 * 维度权重)
```

当前五个维度权重相同：

```text
开放性 20%
尽责性 20%
外向性 20%
宜人性 20%
情绪稳定性 20%
```

### Q24：高契合维度和差异维度怎么判断？

A：系统先计算每个维度的单维契合度，然后排序：

- 契合度最高的两个维度作为“高契合维度”。
- 契合度最低的两个维度作为“差异较大维度”。

之后根据这些维度生成相处建议。

### Q25：为什么双人适配要用邀请码？

A：因为性格画像属于个人隐私数据。邀请码代表用户授权，避免任何人随意查看别人的画像。即使授权后，系统也只展示维度差异和建议，不展示对方完整答题内容。

### Q26：适配分高是否代表两个人一定适合？

A：不能这样理解。适配分只是基于性格维度差异的辅助参考，不是绝对判断。系统给出的是“沟通和相处建议”，不是对关系做定论。

## 七、隐私与安全

### Q27：你们如何保护用户隐私？

A：主要有几层设计：

- 个人报告需要登录后查看。
- 双人适配必须通过邀请码授权。
- 匹配报告不展示完整答题，只展示维度差异。
- 用户可以撤销匹配授权。
- 用户可以清除自己的画像数据。
- 后端接口设计上保留鉴权，不能随意通过 userId 查看他人数据。

### Q28：密码是明文存储吗？

A：不是。后端使用 Spring Security 相关机制，密码不会以明文方式直接存储。答辩时可以强调：密码安全是后端基础安全要求之一，系统不会把明文密码暴露给前端。

### Q29：为什么分享报告只展示摘要？

A：分享报告用于公开展示，所以只展示摘要维度和解释，不展示完整答题内容。这是为了在可分享性和隐私保护之间取得平衡。

## 八、后台与数据管理

### Q30：后台管理有什么作用？

A：后台用于管理员查看和维护系统数据，包括：

- 用户列表和用户状态。
- 题库内容。
- 推荐项。
- 推荐规则。
- 统计概览。

这样系统不是纯前端 Demo，而是有基本运营管理能力。

### Q31：题库以后能不能替换？

A：可以。当前题库已经封装为数据模块，后续也可以通过后台题库接口替换正式题目。前端测试页面只关心题目结构，不关心题目来源，因此迁移成本较低。

### Q32：推荐规则以后能不能调整？

A：可以。推荐权重和推荐项都做了独立封装。当前权重是 40%、40%、20%，后续可以根据真实用户反馈调整权重，或者在后台管理推荐规则。

## 九、测试与构建

### Q33：你们做了哪些测试？

A：前端有 Vitest 测试，覆盖：

- 首页渲染。
- 登录页渲染。
- 工作台数据展示。
- 后台管理页面。
- 产品核心逻辑，包括计分、推荐反馈降权、双人匹配。

后端有 Maven 测试，覆盖：

- 数据种子。
- 接口产品化能力。
- 安全配置。
- 计分、推荐、匹配和报告服务。

### Q34：前端构建是否通过？

A：在本机 PowerShell 中 `npm run build` 可以完成生产构建，生成 `frontend/dist/`。构建输出中可能会提示 chunk 大于 500 KB，这是 Vite 的体积警告，不是构建失败。主要原因是 ECharts 这类图表库体积较大，后续可以通过动态导入或手动分包优化。

### Q35：测试里如果遇到 RouterLink 警告怎么办？

A：Vue 单元测试中如果单独挂载页面组件，需要 stub 掉 `RouterLink`。项目测试已经对相关页面补了 stub，避免测试时因为没有真实路由上下文而报警。

## 十、部署与运行

### Q36：项目怎么部署？

A：当前临时发布链路是：

```text
Cloudflare Tunnel -> Caddy :8090 -> 前端 dist + /api 代理 -> 后端 :8080
```

前端由 Vite 构建成静态文件，Caddy 负责托管静态文件和代理 API，Cloudflare Tunnel 负责把本机服务暴露到公网域名。

### Q37：为什么用 Caddy 和 Cloudflare Tunnel？

A：课程展示阶段没有必要马上购买云服务器。Cloudflare Tunnel 可以把本机服务安全地映射到公网域名，Caddy 负责本机静态文件和 API 代理。这样成本低，部署快，适合临时展示。

### Q38：Docker 在项目里负责什么？

A：Docker Compose 用于启动 MySQL、Redis 和后端服务，保证运行环境一致。这样换一台机器时，不需要手动安装和配置数据库。

## 十一、项目亮点

### Q39：这个项目最大的亮点是什么？

A：亮点有三个：

- 闭环完整：测试、报告、推荐、反馈、匹配、个人中心都能串起来。
- 推荐可解释：不只给结果，还说明为什么推荐。
- 隐私意识：双人适配需要授权，不展示完整答题内容。

### Q40：相比普通性格测试有什么区别？

A：普通性格测试通常停留在“你是什么类型”。我们的系统把画像继续用于实际生活场景：吃什么、去哪玩、适合什么社交方式、两个人如何相处，所以产品感更完整。

### Q41：为什么说它是软件工程项目，而不是静态页面？

A：因为它包含完整的软件工程结构：

- 前后端分离。
- 路由和组件拆分。
- service 层封装数据访问。
- utils 层封装计分、推荐、匹配算法。
- 后端接口和数据库模型。
- 后台管理。
- 测试用例。
- 部署脚本和部署文档。

## 十二、局限与后续改进

### Q42：当前项目有什么不足？

A：主要不足是：

- 当前推荐算法还是规则模型，不是真正的机器学习。
- 前端部分产品状态仍使用 localStorage，后续需要完全接入后端。
- 推荐内容还偏样例，正式上线前需要更专业的内容运营。
- ECharts 体积较大，构建后 JS chunk 偏大。

### Q43：后续如何改进推荐算法？

A：可以从三步改进：

1. 收集更多真实用户反馈。
2. 根据反馈统计调整标签权重和推荐项基础分。
3. 后续可以引入协同过滤或轻量机器学习模型，但当前课程版本优先保证可解释性和可维护性。

### Q44：后续如何改进题库？

A：可以让后台支持题目编辑、维度配置和启停状态；同时邀请更多用户试测，根据题目区分度和用户反馈筛选题目，逐步形成更稳定的正式题库。

### Q45：后续如何优化前端？

A：可以继续做：

- ECharts 动态导入，降低首屏 JS 体积。
- 移动端细节适配。
- 图表动画和报告导出。
- 更完整的后台编辑表单。
- 更精细的错误重试和加载状态。

## 十三、答辩时的简短总述

如果老师让你用一分钟介绍项目，可以这样说：

> 我们做的是一个叫「性格雷达·生活指南」的 Web 应用。用户完成性格、饮食、旅游和社交倾向测试后，系统会把 1-5 分量表归一化为 0-100 的维度分数，并用 ECharts 生成雷达图报告。推荐模块使用规则加权模型，综合性格匹配、场景偏好和用户历史反馈，给出可解释的餐饮、旅游、社交建议。双人适配使用维度差异模型，通过邀请码授权比较双方画像，只展示契合度、差异和相处建议，不暴露完整答题内容。项目包含前后端、后台管理、测试、部署脚本和公网临时发布链路，目标是把普通性格测试扩展成一个有完整用户闭环的生活建议产品。
