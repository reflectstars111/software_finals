## 1. 文档目的

本文档用于定义“性格雷达·生活指南个性化推荐软件”的功能需求、非功能需求、业务规则、接口约束、数据管理要求和运行环境要求，作为后续概要设计、详细设计、编码实现、测试用例编写、系统验收与项目评审的统一依据。

预期读者包括项目经理、软件开发工程师、测试工程师、UI 设计师、系统管理员、项目审核人员和指导老师。

## 2. 项目概述

### 2.1 项目背景

18-35 岁年轻群体对自我认知、性格探索、个性化决策和生活效率的需求持续提升，性格测试、心理测评和个性化推荐类工具逐渐成为高频互联网服务。现有同类工具普遍存在性格维度单一、可视化效果弱、推荐场景有限、缺少反馈优化、双人适配能力不足、重测试轻应用等问题。

本系统以大五人格理论为基础，通过多维度性格测试、雷达图可视化和生活场景推荐，将抽象人格特征转化为可理解、可使用的生活建议。

### 2.2 软件定位

本系统定位为轻量级、移动端优先的 Web 应用，面向普通用户提供性格测评、人格画像、生活推荐、推荐反馈、双人性格适配和报告分享能力，同时为管理人员提供题库、推荐规则、反馈记录和基础数据维护能力。

当前版本为课程实践项目，系统独立运行，不依赖第三方业务平台，不接入第三方社交登录、支付、广告推荐引擎或即时通讯能力。

### 2.3 项目目标

业务目标：

- 打造轻量化、低门槛的性格分析与生活推荐工具。
- 帮助用户进行自我认知，并为饮食、穿搭、旅行、生涯规划等生活场景提供参考。
- 上线后累计用户量目标不少于 5000 人。

技术目标：

- 前端使用 Vue3 与 ECharts，后端使用 Spring Boot，数据库使用 MySQL 8.0。
- 支持不少于 100 名用户同时在线操作。
- 常规页面与核心业务响应时间不超过 2 秒。
- 用户敏感数据加密存储，满足隐私保护要求。

## 4. 用户与角色

### 4.1 普通用户

普通用户是系统的核心服务对象，主要为 18-35 岁青年群体。该类用户通常熟悉智能手机、平板和 PC 浏览器的日常操作，无需具备计算机专业知识。使用场景以碎片化时间为主，单次测评和报告查看时长通常为 5-15 分钟。

普通用户主要诉求包括：

- 完成基础性格、饮食偏好和旅游偏好测试。
- 查看个人综合画像报告和性格雷达图。
- 获取生活场景推荐。
- 对推荐内容进行反馈。
- 发起或参与双人性格适配。
- 管理个人资料、头像和历史测试记录。

### 4.2 管理人员

管理人员负责系统后台运维和基础数据维护，需要具备基本计算机专业背景，熟悉 Vue3、Spring Boot、MySQL、RESTful API 和基础服务器运维操作。

管理人员主要职责包括：

- 维护测试题库。
- 调整推荐规则和权重映射。
- 查看用户反馈记录。
- 管理系统基础数据。
- 查看日志并排查异常。

### 4.3 特殊用户

特殊用户包括心理咨询师、职场规划人员等，可借助系统生成的性格报告辅助开展咨询、分析和用户研判。

## 5. 范围、假定与约束

### 5.1 功能范围

当前版本包含：

- 手机号注册、账号密码登录、个人资料编辑和头像管理。
- 基础性格测试、饮食偏好测试、旅游偏好测试。
- 性格维度得分计算、雷达图渲染和综合画像报告生成。
- 餐饮、旅游、穿搭、生涯规划等生活场景推荐。
- 推荐内容的“喜欢 / 一般 / 不喜欢”反馈。
- 基于历史测试数据的双人性格适配。
- 报告图文海报生成与分享。
- 管理端题库维护、推荐规则维护、反馈查看、日志查看和基础数据管理。

当前版本不包含：

- 用户付费功能。
- 广告投放功能。
- 第三方社交平台快捷登录。
- 第三方支付。
- 即时通讯。
- 独立 PC 客户端。
- 复杂机器学习推荐算法。

### 5.2 开发假定

- 项目开发期间网络环境稳定，无长期断网和服务器长期宕机情况。
- ECharts、加密工具等开源组件可正常使用，且无版权纠纷。
- 项目需求无重大变更，仅允许小范围优化调整。
- 项目开发人员无大规模流动，能够保证开发进度。

### 5.3 开发约束

- 项目为高校课程实践项目，总开发周期约 14 周，需按教学日历完成设计、开发、测试和验收。
- 项目无商业采购经费，必须基于开源免费技术栈实现。
- 固定技术栈为 Vue3、ECharts、Spring Boot、MySQL 8.0。
- 推荐策略初期采用预设规则引擎和标签匹配逻辑，不引入复杂机器学习模型。
- 系统涉及用户性格和偏好数据，必须遵守《中华人民共和国个人信息保护法》，采用数据脱敏、加密存储和访问控制。
- 系统无需专用硬件，应适配普通手机、平板和电脑设备。

## 7. 核心用例

### 7.1 查看综合画像报告

前置条件：

- 普通用户已登录。
- 用户至少完成基础性格测试。

基本事件流：

1. 用户在客户端点击“查看我的报告”。
2. 系统调用测试控制器，提取该用户的测试答题记录。
3. 系统计算多维度人格特征得分向量。
4. 系统生成性格雷达图。
5. 系统生成各维度文字解析，并匹配穿搭、学习、生涯规划等文本建议。
6. 前端整合雷达图和文字报告，向用户展示综合画像报告。

输出结果：

- 性格维度得分。
- 性格雷达图。
- 综合文字画像。
- 多场景生活建议。

### 7.2 查看双人适配报告

前置条件：

- 发起方用户已登录。
- 发起方已绑定或输入好友 ID。
- 双方均存在历史测试数据。

基本事件流：

1. 用户在社交与关系适配模块中点击“查看双人适配报告”。
2. 系统提取双方既有测试画像数据。
3. 系统禁止唤起重复做题流程。
4. 系统根据多维度差异值和规则权重计算适配分数。
5. 系统生成冲突预警、互补优势和相处建议。
6. 页面展示双人适配报告。

输出结果：

- 契合度得分。
- 潜在分歧点。
- 互补优势。
- 相处建议。

### 7.3 提交推荐评价

前置条件：

- 用户正在浏览餐饮、旅游或其他场景推荐列表。

基本事件流：

1. 用户对推荐卡片点击“喜欢”“一般”或“不喜欢”。
2. 前端异步发送用户 ID、推荐项 ID、评价等级和可选文字反馈。
3. 后端推荐引擎记录反馈数据。
4. 系统修正用户对应场景标签权重。
5. 系统返回反馈成功提示，并影响后续推荐结果。

输出结果：

- 反馈提交结果。
- 更新后的用户推荐偏好权重。
- 后续优化后的推荐内容。

## 8. 数据与业务规则

### 8.1 输入要求

- 手机号必须为 11 位纯数字。
- 密码长度为 6-16 位字符，禁止特殊非法字符。
- 测试输入采用统一标准化答题格式。
- 反馈评价分为喜欢、一般、不喜欢。
- 文字反馈长度限制为 0-100 字符。

### 8.2 输出要求

- 图表输出为彩色性格雷达图，需标注维度名称和具体得分。
- 报告输出应采用图文结合形式，排版整洁并适配手机浏览。
- 分享输出支持 PNG 高清图片和加密短链接。
- 异常输出应通俗易懂，明确告知异常原因和解决方式。

### 8.3 数据管理要求

- 核心数据表应至少包含用户表、测试表、推荐表、反馈表、适配表。
- 初期存储数据量预计不超过 10GB。
- 单用户全部数据占用空间不超过 1MB。
- 每年数据增长量按原有数据约 2 倍预估，数据库需预留扩容空间。
- 正常用户数据永久保存。
- 用户注销账号后，相关数据应在 7 天内彻底清除且不可恢复。
- 数据库应每日自动备份。

### 8.4 核心实体

| 实体 | 职责 |
| --- | --- |
| User | 维护用户账号信息、基础资料和可选地理位置坐标。 |
| Question | 存储测试题目、测试类型和选项维度权重。 |
| TestResult | 存储用户完成测试后的多维特征得分，是推荐和双人适配的数据基础。 |
| Feedback | 记录用户对推荐项的喜欢、一般、不喜欢等主观评价。 |
| CompatibilityReport | 存储双人适配结果，包括总分、分歧预警点和相处建议。 |

### 8.5 分析类职责

| 层次 | 类/组件 | 职责 |
| --- | --- | --- |
| 表现层 | TestPage | 渲染测试题目，捕获用户答题数据。 |
| 表现层 | IntegratedReportPage | 展示雷达图、多场景报告，并提供反馈与分享入口。 |
| 表现层 | DualMatchPage | 输入或扫描好友信息，展示双人适配结果。 |

| 业务层 | RecommendationEngine | 根据测试结果和反馈权重匹配推荐项，并动态修正规则。 |
| 业务层 | MatchController | 提取双方测试结果，计算适配度并生成适配报告。 |

## 9. 接口需求

### 9.1 通用接口规范

- 通信协议：生产环境采用 HTTPS，开发环境可兼容 HTTP。
- 接口风格：遵循 RESTful API 设计规范。
- 数据格式：统一使用 JSON。
- 字符编码：统一使用 UTF-8。
- 请求方式：GET 查询、POST 提交、PUT 修改、DELETE 删除。
- 前端请求组件：使用 Axios 异步请求，实现无刷新数据交互。

### 9.2 内部业务接口

| 接口类型 | 主要能力 |
| --- | --- |
| 用户接口 | 注册、登录、个人信息修改、密码加密、权限校验 |
| 测试接口 | 题目拉取、答案提交、分数计算、历史记录查询 |
| 推荐接口 | 性格数据匹配、场景推荐、反馈权重修改 |
| 适配接口 | 好友数据调取、双人适配计算、适配报告生成 |
| 后台接口 | 题库维护、规则修改、日志查询、数据统计 |

### 9.3 第三方组件接口

## 12. 控制与异常处理

### 12.1 控制方式

人为手动控制：

- 用户通过点击、滑动、输入、提交等触控或鼠标操作触发业务流程。
- 管理人员通过后台按钮完成题库、规则和数据维护。
- 操作人员通过 URL 访问、刷新和跳转控制页面访问。

系统自动控制：

- 后端定时备份数据库和清理无效缓存。
- 测试分数、适配分值和推荐权重由后台算法自动运算。
- 前端页面加载后自动渲染图表并初始化用户数据。
- 服务器进程后台常驻，异常时自动重启。

### 12.2 控制信号来源

- HTTP 请求信号：用户界面操作转化的网络请求。
- Vue 生命周期钩子信号：页面挂载、更新、销毁时触发数据初始化或清理。
- 数据库连接信号：连接池自动管控联机、断开和事务回滚。
- 定时任务信号：Spring Task 定时任务用于数据备份和日志清理。

### 12.3 异常控制要求

- 所有异常应统一封装为友好提示。
- 系统不得直接向用户暴露代码报错、异常堆栈、源码路径或数据库异常信息。
- 权限非法访问应自动拦截并跳转登录页面。
- 所有故障不得泄露用户隐私数据。
- 故障恢复时间应不超过 30 分钟。
- 故障日志应自动留存，便于运维排查。

## 13. 验收要点

项目验收时至少应覆盖以下内容：

- 用户可完成注册、登录、个人资料维护和历史记录查看。
- 用户可完成三类测试并生成综合画像报告。
- 雷达图可正常渲染，维度名称和分数显示清晰。
- 系统可基于测试结果生成生活推荐。
- 用户推荐反馈可保存，并影响后续推荐权重。
- 用户可通过好友 ID 或分享链路生成双人适配报告。
- 管理人员可进入后台维护题库、推荐规则和查看反馈。
- 普通用户与管理人员权限隔离有效。
- 核心接口返回 JSON，编码为 UTF-8，前后端交互稳定。
- 常规操作响应时间满足性能要求。
- 移动端页面无严重错位、遮挡或横向滚动问题。
- 敏感数据不明文存储，用户数据不可越权访问。
- 推荐为空、登录失败、网络异常、好友数据不完整等异常场景均有友好提示。
- 系统可在课程演示环境完成部署、运行和演示。
