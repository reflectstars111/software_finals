# Codex 改进任务 Prompt：完善「性格雷达·生活指南」Web 产品

## 一、项目背景

当前项目名称为「性格雷达·生活指南」，是一个大学软件工程课程大作业项目，定位为：

> 一个集性格测试、个性化生活推荐、双人适配分析和个人数据管理于一体的 Web 移动端应用。

当前已规划或已存在的核心页面包括：

```text
/tests
/report
/recommendations
/match
/profile
```

项目希望从「课程 Demo」进一步完善为一个更像真实软件产品的 Web 应用。
请基于现有代码进行产品化改造，重点不是盲目增加页面数量，而是完善用户路径、页面信息架构、交互状态、数据闭环、推荐解释性和隐私安全设计。

---

## 二、整体产品目标

请将当前系统改造成一个完整闭环产品：

```text
进入网站
↓
了解产品价值
↓
开始测试
↓
完成性格/饮食/旅游/社交倾向测评
↓
生成综合画像报告
↓
基于报告获得饮食/旅游/社交推荐
↓
用户对推荐进行反馈
↓
系统根据反馈优化后续推荐
↓
用户进行双人适配分析
↓
用户在个人中心管理测试记录、推荐反馈、匹配授权和隐私数据
```

最终用户应该明确感受到：

> 这个系统不是单纯做性格测试，而是能根据我的性格画像，帮我做饮食、旅游和社交方面的生活决策。

## 五、`/tests` 测试页面改造要求

`/tests` 是系统的数据入口，需要从普通问卷改造成正式测评体验。

### 1. 测试分类

请将测试拆分为以下模块：

```text
基础性格测试
饮食偏好测试
旅游偏好测试
社交倾向测试
```

可以先用模拟题库实现，但代码结构要支持后续从后端题库读取。

### 2. 题目选项

所有题目建议统一使用 5 级量表：

```text
非常不同意
不太同意
一般
比较同意
非常同意
```

每个选项建议映射为 1-5 分。

### 3. 页面功能要求

请增加：

- 当前测试进度；
- 总进度条；
- 当前题目编号，例如 `6 / 30`；
- 上一题 / 下一题；
- 防漏答校验；
- 提交前确认；
- 提交中 loading；
- 提交失败提示；
- 提交成功后跳转 `/report`。

### 4. 计分规则

请设计一个清晰可维护的计分结构。
建议每道题包含：

```ts
{
  id: string
  category: 'personality' | 'food' | 'travel' | 'social'
  dimension: string
  text: string
  reverse?: boolean
}
```

提交时生成用户画像，例如：

```ts
{
  openness: 82,
  conscientiousness: 61,
  extraversion: 76,
  agreeableness: 70,
  neuroticism: 42,
  foodAdventure: 80,
  foodSocial: 75,
  travelAdventure: 68,
  travelPlanning: 55,
  socialEnergy: 78
}
```

### 5. 空状态和异常状态

如果题库加载失败，显示：

```text
题目加载失败，请稍后重试。
```

如果用户未答完，显示：

```text
你还有 X 道题未完成，请完成后再生成报告。
```

## 七、`/recommendations` 推荐页面改造要求

`/recommendations` 是产品差异化核心，必须体现“个性化”和“可解释”。

### 1. 推荐分类

至少支持三类推荐：

```text
餐饮推荐
旅游推荐
社交推荐
```

可以使用 Tab 切换。

### 2. 推荐卡片字段

每个推荐卡片至少包含：

```text
推荐名称
推荐类型
匹配度
推荐标签
推荐理由
适合你的原因
反馈按钮：喜欢 / 一般 / 不喜欢
```

示例：

```text
推荐：东南亚融合餐厅
类型：餐饮
匹配度：88%

推荐理由：
你的开放性较高，系统判断你更愿意尝试新口味；同时你的社交倾向较强，因此优先推荐适合朋友聚餐的餐厅。

反馈：
喜欢 / 一般 / 不喜欢
```

### 3. 推荐算法说明

请实现或至少封装一个清晰的规则加权推荐模型：

```text
推荐得分 = 性格匹配分 × 40%
        + 场景偏好分 × 40%
        + 历史反馈分 × 20%
```

代码中请将权重抽离为常量，方便修改。

示例：

```ts
const RECOMMENDATION_WEIGHTS = {
  personality: 0.4,
  preference: 0.4,
  feedback: 0.2
}
```

### 4. 推荐理由生成

每条推荐必须能根据用户画像生成解释，例如：

```text
因为你的开放性较高
因为你偏好轻松社交场景
因为你不喜欢高强度行程
因为你更适合计划明确的旅行路线
```

推荐理由不能完全写死，要尽量和用户画像字段关联。

### 5. 反馈机制

用户点击：

```text
喜欢
一般
不喜欢
```

后需要：

- 显示反馈成功提示；
- 保存反馈记录；
- 更新本地推荐权重或重新排序；
- 至少让用户看到推荐排序发生变化，或者提示后续推荐已优化。

反馈成功文案：

```text
已收到反馈，后续将减少相似类型推荐。
```

或：

```text
已收到反馈，后续将优先推荐相似内容。
```

### 6. 无画像数据状态

如果用户还没有完成测试，显示：

```text
请先完成测试，系统将根据你的画像生成个性化推荐。
```

并提供按钮：

```text
去完成测试
```

## 八、`/match` 双人适配页面改造要求

`/match` 是项目亮点，不能只展示一个匹配百分比。
需要做成完整的关系分析报告。

### 1. 授权机制

请实现基础邀请码机制，避免用户直接查看他人画像。

功能包括：

```text
生成我的匹配邀请码
输入对方邀请码
双方授权后生成适配报告
```

如果暂时不接后端，可以先用本地模拟数据实现，但 UI 和数据结构要预留。

### 2. 页面结构

```text
匹配入口区
生成/输入邀请码
双方基础信息
双人雷达图对比
总体契合度
高契合维度
差异较大维度
潜在冲突提醒
相处建议
隐私说明
```

### 3. 计算逻辑

建议使用加权差异模型：

```text
维度差异 = abs(用户A维度分 - 用户B维度分)
单维契合度 = 100 - 维度差异
总体契合度 = Σ(单维契合度 × 维度权重)
```

维度权重示例：

```ts
const MATCH_WEIGHTS = {
  openness: 0.2,
  conscientiousness: 0.2,
  extraversion: 0.2,
  agreeableness: 0.2,
  emotionalStability: 0.2
}
```

### 4. 结果解释

请不要只显示：

```text
契合度 82%
```

必须显示解释：

```text
高契合维度：
你们在开放性和外向性上较接近，适合一起尝试新鲜活动、短途旅行和朋友聚会。

潜在差异：
你们在尽责性上差异较大，一方更喜欢提前规划，另一方更偏向随性安排。

相处建议：
在旅行或聚餐前，建议提前确认预算、时间和路线，同时保留一定自由调整空间。
```

### 5. 隐私说明

请在页面底部显示：

```text
该适配报告基于双方已授权的历史测试数据生成，不会展示对方完整答题内容，仅展示维度差异和相处建议。
```

### 6. 状态处理

请处理以下状态：

```text
未完成测试
未输入邀请码
邀请码错误
等待对方授权
双方数据不足
匹配成功
```
